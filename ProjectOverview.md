# Contextual Agentic Retrieval — Architecture & Roadmap

---

## What This Is

A document research system where an AI agent doesn't just search — it **reasons** its way through a corpus the same way a human researcher would: starting broad, narrowing down, following references, and reconstructing relationships between sections. Built for corpora that are too large for pure agentic search and too relational for pure RAG.

There is no existing off-the-shelf system that combines everything described here. GraphRAG, RAPTOR, and Late Chunking each exist independently. This architecture combines the navigational structure of RAPTOR, the relationship graph of GraphRAG, and the contextual embedding depth of Late Chunking, unified under an agentic control loop. That specific combination does not exist as a package.

---

## The Secret Recipe

### Ingredient 1 — Late Chunking (the foundation)

Normal RAG embeds each chunk in isolation. A chunk that says "subject to the exceptions in the following article" has no idea what the following article says — that meaning is lost the moment you split the document.

Late chunking reverses the order: embed the **entire document first**, then split. Because the embedding model reads the full document before splitting, every chunk's vector carries awareness of the surrounding context. Article 89 "knows about" Article 134 even in its own vector. This makes individual chunk retrieval dramatically more precise without changing the storage structure.

**The key model requirement:** a long-context embedding model — `bge-m3` or `jina-embeddings-v3` — that can hold an entire document in its attention window before chunking begins.

---

### Ingredient 2 — Hierarchical Index (the navigation map)

The corpus gets indexed at three levels simultaneously:

```
Level 1 — one summary per document
           "This document covers X, Y, Z across N sections"

Level 2 — one summary per section/chapter
           "Section 4 defines obligations for entities classified under Article 3"

Level 3 — individual chunks (late-chunked, contextually aware)
           actual content, with metadata: source, page range, section ID, parent ID
```

The Level 1 and 2 summaries are generated by the LLM during indexing. They are not retrieved for the final answer — they are used exclusively for **navigation**, so the agent knows which documents and sections to target before reading any actual content. This is the core mechanism that avoids loading irrelevant material.

---

### Ingredient 3 — Relationship Graph (the connective tissue)

Two types of edges, extracted during indexing:

**Explicit edges** — structural cross-references found in the text itself:
```
"see Article 134" → Article_89 ──references──→ Article_134
"as defined in Schedule B" → Clause_22 ──defined_by──→ Schedule_B
```

**Implicit edges** — computed from embedding similarity at index time:
```
chunk_A and chunk_B have cosine similarity > 0.85
→ Section_4_doc_X ──topically_linked──→ Section_7_doc_Y
```

These edges are stored as a lightweight adjacency structure alongside the vector index. They are what make cross-document reasoning possible without loading full documents — the agent can traverse a relationship instead of searching for it.

---

### Ingredient 4 — Agentic Control Loop (the brain)

The existing FsExplorer event loop — unchanged. What changes is the toolset available to the agent:

| Tool | What it does |
|------|-------------|
| `hier_search(query, level)` | Navigate the hierarchy: L1 finds docs, L2 finds sections, L3 finds chunks |
| `follow_references(chunk_id)` | Traverse explicit graph edges — jump to referenced sections directly |
| `find_related(chunk_id)` | Traverse implicit edges — surface semantically linked sections across documents |
| `expand_context(chunk_id, window)` | Return a chunk plus its N neighbors — restores narrative continuity on demand |
| `get_section(file, page_start, end)` | Raw filesystem access for small/new unindexed files |

The agent decides which tool to call at each step. It starts broad (L1), narrows down (L2→L3), follows references when found, expands context when a chunk is cut off mid-thought. The LLM never sees a full document — only the sections it explicitly navigated to.

## How It All Flows Together

```
Query: "Find all points related to X across all documents"

[L1 search] → "doc_A and doc_C are relevant, others can be ignored"
     ↓
[L2 search on doc_A] → "Section 3.2 and Section 7.1 are the relevant sections"
     ↓
[L3 search on Section 3.2] → "4 chunks returned. chunk_89 says: see Article 134"
     ↓
[follow_references(chunk_89)] → "Article 134 retrieved directly from graph"
     ↓
[expand_context(Article_134)] → "chunk was mid-sentence — neighbors added, continuity restored"
     ↓
[find_related(chunk_89, cross_document=True)] → "doc_C Section 4.1 is topically linked"
     ↓
[L3 search on doc_C Section 4.1] → "3 more chunks retrieved"
     ↓
[STOP] → synthesize answer from ~12 targeted chunks across 2 documents
```

Token usage: ~8,000. Equivalent pure agentic search on the same documents: ~400,000+.

## What's Genuinely Innovative

**`expand_context` is novel.** No existing system does dynamic, agent-controlled narrative window expansion. GraphRAG fetches full parent sections. RAPTOR retrieves full tree branches. This approach restores continuity only where the agent identifies it's needed, keeping token usage minimal.

**Structural edges as first-class graph citizens.** GraphRAG builds edges from entity extraction (people, places, concepts). For documents with explicit internal references — laws, contracts, technical manuals — the most important relationships are the structural ones: "see Article X", "per Schedule B". Extracting these as explicit edges during indexing and making them directly traversable is the primary thing that separates this architecture from GraphRAG for document-heavy use cases.

**Late chunking + hierarchy combined.** Late chunking produces contextually rich chunks. Hierarchical indexing provides navigational structure. Neither existing system combines both. Separately they each solve half the problem. Together they solve it fully: the agent navigates precisely and the chunks it retrieves carry deep contextual meaning.

## Logical Challenges (Honest)

**Indexing cost is real.** Generating LLM summaries at Level 1 and Level 2 during indexing costs tokens. A corpus of 100 documents with average 20 sections each means ~2,000 LLM calls just for summaries. This is a one-time offline cost, but it's not free. Mitigation: cache aggressively, re-index only changed files.

**Implicit edge quality depends on threshold tuning.** Setting cosine similarity too low produces noisy connections. Too high and you miss real relationships. The right threshold varies by domain and corpus. This requires calibration per use case.

**Late chunking requires a long-context embedding model.** `bge-m3` supports up to 8,192 tokens. A 403-page document is far beyond that. The practical solution is to apply late chunking at the section level rather than the full document level — embed each chapter/section as a whole before splitting it into chunks. You lose some cross-section awareness but retain intra-section coherence, which is the most important.

**The graph is only as good as the reference extraction.** Regex-based extraction catches "see Article X" but misses implicit dependencies ("the definition established earlier" with no explicit reference). This is a known limitation of structural edge extraction and is where a more sophisticated NLP parsing step — or even a small LLM call during indexing — would improve quality in future iterations.

## MVP Roadmap

---

### MVP 1 — Contextual Agentic Search (current system + page ranges)
*Goal: make what you have actually work on large files*

The existing FsExplorer loop, with two additions:
- `parse_file(file, page_start, page_end)` — agent reads large documents in sections, 50 pages at a time
- OCR disabled in Docling (already done)

The agent navigates large documents by requesting sections sequentially or by reasoning about which page range is likely to contain the answer based on the preview. No vector index, no graph. Pure agentic reasoning, now unblocked by memory limits.

**What it delivers:** works on any file size, no new infrastructure, citations from actual document sections.
**Limitation:** still linear navigation, still token-heavy for very large corpora.

---

### MVP 2 — Flat Semantic Search Layer
*Goal: add vector retrieval as a parallel tool alongside filesystem tools*

Add ChromaDB. Build a simple indexing script: Docling → fixed-size chunks (512 tokens) → standard embeddings (`multilingual-e5-small`) → stored with metadata (source file, page, section title if detectable).

Add two tools to the agent:
- `semantic_search(query, limit)` — returns top N chunks with source metadata
- `get_section(file, page_start, page_end)` — filesystem fallback

The agent now has a choice: use `semantic_search` for targeted lookup, use filesystem tools for structural navigation. No hierarchy yet, no graph. The agent decides based on the query type.

**What it delivers:** handles large corpora fast, locatable answers retrieved in milliseconds, token usage drops dramatically.
**Limitation:** flat retrieval — no navigation structure, no cross-references, chunks lose context at boundaries.

---

### MVP 3 — Hierarchical Navigation + Late Chunking
*Goal: replace flat retrieval with structured navigation and contextually rich chunks*

Replace standard chunking with late chunking at the section level. Replace the flat ChromaDB collection with three collections: L1 (document summaries), L2 (section summaries), L3 (late-chunked content).

Replace `semantic_search` with `hier_search(query, level)`. The agent now navigates top-down: identify relevant documents → identify relevant sections → retrieve specific chunks. Add `expand_context(chunk_id, window)` for narrative continuity restoration.

LLM-generated summaries for L1 and L2 are produced during the indexing pipeline run.

**What it delivers:** precise navigation across large corpora, contextually aware chunks, narrative continuity on demand, answer quality approaches pure agentic search.
**Limitation:** no explicit cross-reference traversal yet — the agent still has to search for referenced sections rather than jumping to them directly.

---

### MVP 4 — Relationship Graph + Reference Traversal
*Goal: make cross-references first-class, enable true multi-document reasoning*

During indexing, parse explicit cross-references from document text ("see Article X", "per Schedule B", "as defined in Section 3"). Store as an adjacency graph alongside the vector index.

Compute implicit edges between chunks with cosine similarity above threshold. Store the highest-confidence cross-document links.

Add tools:
- `follow_references(chunk_id)` — traverse explicit edges, return referenced chunks directly
- `find_related(chunk_id, cross_document)` — traverse implicit edges, surface topically linked sections

The agent now has the full toolset. It navigates via hierarchy, follows explicit references without re-searching, and can surface cross-document relationships it didn't explicitly search for.

**What it delivers:** the complete architecture. Multi-document reasoning that matches pure agentic search quality at a fraction of the token cost. Cross-reference traversal that RAG fundamentally cannot do.

---

### MVP 5 — Production Hardening
*Goal: make the system reliable, observable, and maintainable*

- Incremental indexing: only re-index files that have changed (hash-based detection)
- Index health endpoint: `GET /api/index/status` — shows which files are indexed, when, coverage
- Query observability: log which tools the agent called, which chunks were retrieved, total tokens used per query
- Chunk quality scoring: flag sections where late chunking was degraded (documents with poor structure)
- Multi-language embedding model selection: auto-select embedding model based on detected document language

**What it delivers:** a system you can operate, monitor, and trust in production. The intelligence is in MVP 4; MVP 5 makes it reliable enough to depend on.

## Pipeline
Document
    ↓
Docling → structured markdown
    ↓
Structure Parser
    ├── extract headings → hierarchy skeleton (free)
    ├── detect explicit cross-references → graph edges (free, regex)
    └── flag sections with missing or uninformative headings
    ↓
For flagged sections only:
    LLM call on first 200 words → one-sentence summary
    ↓
For document level:
    LLM call on table of contents / heading list → paragraph summary
    ↓
Late chunking on each section:
    embed full section → split → store contextual chunk vectors
    ↓
Compute implicit edges:
    cosine similarity between chunk vectors → store high-confidence links
    ↓
ChromaDB: L1 collection, L2 collection, L3 collection
Graph: adjacency dict stored as JSON alongside the index
